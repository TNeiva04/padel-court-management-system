# Integração e Hospedagem da Aplicação no Render

## Objetivo de Criar a Instância no Render

O principal objetivo de criar uma instância de base de dados PostgreSQL no Render foi colocar a aplicação num ambiente online, para que pudesse ser acedida pela internet, garantindo que o sistema funcione de forma estável fora do computador onde foi desenvolvida. O Render é uma plataforma PaaS (Platform as a Service) que facilita a implementação de aplicações e bases de dados, oferecendo uma infraestrutura gerida, com suporte a containers Docker, bases de dados e serviços web.

Ao usar o Render, conseguimos:

- Ter uma base de dados PostgreSQL alojada na cloud, com alta disponibilidade.
- Colocar a aplicação web num serviço gerido, o que torna o processo de deploy mais simples.
- Garantir que a aplicação esteja acessível a partir de qualquer lugar, sem depender do computador onde foi feita.
- Facilitar a gestão das variáveis de ambiente e outras configurações necessárias para a aplicação.


## Porquê que foi preciso mudar a URL de ligação à base de dados

Quando estávamos a desenvolver a aplicação no nosso computador, usávamos uma URL para ligar à base de dados local, por exemplo:

```
" setURL("jdbc:postgresql://localhost/postgres?user=postgres&password=1234")"
```

Esta URL liga-se a uma base de dados que está a correr no nosso computador, o que é suficiente para testar e desenvolver.

Mas quando passámos a aplicação para o Render, tivemos de mudar esta URL para a que o Render nos deu, que aponta para a base de dados na cloud, por exemplo:

```
setURL("jdbc:postgresql://dpg-d0mt77umcj7s739kij30-a.frankfurt-postgres.render.com:5432/db_ls_2425_2_43d_g07?user=db_ls_2425_2_43d_g07_user&password=fMw4BQpn8n85NfJ1gbTcspihkaoXRdVM")
```

Esta mudança foi necessária porque:

- A base de dados local não está acessível a partir do Render.
- O Render dá-nos um endereço seguro para ligar à base de dados.
- A aplicação tem de se ligar à base de dados correta para funcionar com os dados reais.

## O que mudámos no código (router.js e courtsServer.kt)

Para usar a nova URL, tivemos de alterar os ficheiros onde configuramos a ligação à base de dados, como o `router.js` e o `courtsServer.kt`. As mudanças foram:

- Substituir a URL local pela URL do Render.
- Fazer com que a URL seja lida a partir de uma variável de ambiente (`JDBC_DATABASE_URL`), para não termos de pôr as credenciais diretamente no código.
- Ajustar o código para que a porta do serviço seja obtida da variável de ambiente `PORT`, como o Render exige.

Exemplo de como usamos a variável de ambiente no código:

```kotlin
val jdbcUrl = System.getenv("JDBC_DATABASE_URL") ?: "jdbc:postgresql://localhost/postgres?user=postgres&password=1234"
setURL(jdbcUrl)
```

## O que acontece com esta mudança

Com a nova URL, a aplicação passa a:

- Ligar-se a uma base de dados real e acessível a partir de qualquer lugar, permitindo que outras pessoas usem o sistema.
- Garantir que os dados ficam guardados num ambiente seguro e gerido, com backups.
- Facilitar o deploy e a escalabilidade, porque o ambiente de produção está separado do nosso computador.

## Vantagens de usar o Render e mudar a URL

- **Disponibilidade e Escalabilidade:** O Render gere a infraestrutura e permite que a aplicação cresça sem precisarmos de configurar servidores.
- **Facilidade de Deploy:** Usando Docker e o Render, o deploy é mais simples e rápido.
- **Manutenção mais fácil:** O Render trata das atualizações, backups e manutenção da base de dados.
- **Acesso remoto:** A aplicação e a base de dados podem ser acedidas de qualquer lugar, o que é útil para testes e para os utilizadores.
