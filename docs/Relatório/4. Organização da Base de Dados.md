## Entidade-Relação
O diagrama a seguir ilustra o modelo Entidade-Relação
(ER) que reflete as informações geridas pelo sistema. Ele oferece
uma visão geral de como os dados estão estruturados e as relações estabelecidas
entre as entidades principais.

![Diagrama do Modelo](/src/main/sql/LS_DB.jpg)

### Aspectos Destacados
Ao longo do projeto destacamos alguns pontos relevantes
no desenho do modelo conceitual sendo eles:

- **Normalização**: O modelo foi projetado para
  minimizar a redundância e melhorar a integridade dos
  dados, seguindo os princípios de normalização.

- **Atributos Essenciais**: Cada entidade inclui os
  atributos principais necessários para representar o domínio do sistema.

- **Integridade Referencial**: Foram estabelecidas regras
  de integridade para assegurar que as conexões entre entidades
  sejam consistentes.

- **Flexibilidade**: O modelo foi estruturado
  para permitir a evolução do sistema,
  facilitando futuras expansões ou alterações nos requisitos.

---
### Restrições do Modelo Entidade-Relação
O modelo ER foi projetado
considerando algumas restrições importantes para garantir consistência
e conformidade aos requisitos solicitados pelos utentes:

1.**Campos obrigatórios**: Alguns campos foram definidos para serem obrigatórios
e essenciais para o funcionamento das operações do sistema, não podendo conter
valores nulos (ex: e-mails, identificadores únicos)

2.**Limitações na exclusão de dados**: Para evitar
inconsistências, entidades dependentes
de outra tabela só podem ser removidas
seguindo regras específicas ou por
meio de exclusão lógica
(quando aplicável).

3.**Integridade referencial obrigatória**:
Qualquer elemento criado com
dependência direta de outra entidade
(ex: um pedido relacionado a um user)
deverá respeitar a integridade referencial.
Todos os dados são validados antes da persistência no banco de dados.

4.**Associações Exclusivas**: Certos relacionamentos possuem unicidade
(ou seja, a associação é única e exclusiva,
permitindo apenas uma única correspondência
entre as entidades), como nas tabelas
de autenticação ou associações específicas(ex: existe uma relação entre clubes
(_clubs_) e courts (_courts_) através do identificador do clube
)

### Modelo Físico
O modelo físico da base de dados está disponível no arquivo [createSchema.sql](https://github.com/isel-leic-ls/2425-2-LEIC43D-G07/blob/fd79c4166f11bedf605c38c0830318dba4fbb1eb/src/main/sql/createSchema.sql)


#### 1. Relacionamentos entre Tabelas
- A tabela `Club` está relacionada com a tabela `User_Table` através do campo `owner`, que é uma chave estrangeira referenciando o `uid` na tabela `User_Table`. Isso indica que cada clube tem um único proprietário, que é um utilizador registado.
- A tabela `Court` está relacionada com a tabela `Club` através da chave estrangeira `cid`, associando cada campo a um clube.
- A tabela `Rental` possui duas chaves estrangeiras: uma referenciando a tabela `Court` (campo `crid`) e outra referenciando a tabela `User_Table` (campo `uid`). Isso indica que as reservas (Rentals) são feitas por utilizadores para campos específicos.

#### 2. Consistência de Dados
- A tabela `User_Table` tem uma verificação de unicidade no campo `email` e uma validação de formato de email através do `check (email like '%@%')`. Isso ajuda a garantir que os emails sejam únicos e estejam no formato correto.
- O campo `duration_hours` na tabela `Rental` tem uma restrição para garantir que o valor seja superior a 0 e inferior a 24, o que impede entradas inválidas para a duração da reserva.

#### 3. Utilização de `serial` como Tipo para Chaves Primárias
- As tabelas utilizam `serial` como tipo de dado para as chaves primárias (`uid`, `cid`, `crid`, `rid`), o que significa que o valor da chave primária é gerado automaticamente. Isso facilita a inserção de dados sem a necessidade de fornecer valores para estas colunas.

#### 4. Validação de Dados a Nível de Base de Dados
- As validações, como o formato do email e a duração do Rental, ajudam a manter a integridade dos dados no nível da base de dados, evitando entradas inconsistentes ou inválidas.
