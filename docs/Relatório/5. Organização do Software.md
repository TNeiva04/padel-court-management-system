## Organização de Software

O projeto está organizado de forma modular e orientada a interfaces, promovendo a reutilização de código, testabilidade e separação clara de responsabilidades. A estrutura principal encontra-se dividida em packages que refletem os diferentes componentes da aplicação:

### `domain`

Contém os modelos de domínio principais (`User`, `Club`, `Court`, `Rental`). Estas classes representam os conceitos centrais da aplicação e estão livres de dependências externas, permitindo que sejam reutilizadas em qualquer camada, como foi explicado em [Funcionalidades da API](3.%20Funcionalidades%20da%20API.md#funcionalidades-da-api)

### `data`

Responsável pela implementação dos repositórios, encontra-se subdividida em dois packages:

- **`DataMem`**: Contém implementações dos repositórios que utilizam armazenamento em memória, úteis para testes e desenvolvimento sem persistência.
- **`db`**: Contém implementações dos repositórios que interagem com a base de dados PostgreSQL.

Todos os repositórios seguem interfaces comuns (`UserRepository`, `ClubRepository`, etc.), permitindo que a aplicação funcione com qualquer implementação concreta, bastando injetar a desejada.

### `http`

Contém apenas o Server e define os endpoints HTTP da aplicação, organizados por tipo de recurso (`userRoutes`, `clubRoutes`, `courtRoutes`, `rentalRoutes`). Cada conjunto de rotas recebe uma instância dos web APIs respetivos, os quais encapsulam os serviços de negócio.

### `error`

Responsável pelo tratamento de erros e exceções, assegurando um controlo centralizado de respostas HTTP em caso de falha.

### Serviços

Existe um conjunto único de serviços (por exemplo, `UserService`, `RentalService`, etc.) utilizados por ambas as versões dos repositórios (em memória ou base de dados). Estes serviços contêm a lógica de negócio da aplicação e dependem apenas das interfaces dos repositórios, o que garante que a lógica de negócio não precisa de ser alterada quando muda o tipo de persistência.

## Especificação da API (OpenAPI)

A aplicação é documentada com recurso a uma especificação OpenAPI definida no ficheiro [courts.yaml](https://github.com/isel-leic-ls/2425-2-LEIC43D-G07/blob/3e414ca439a017dd3874801bb25ba9a0520d934e/docs/courts.yaml), que descreve de forma clara e formal todos os endpoints disponíveis. Os pontos-chave desta especificação incluem:


**Estrutura dos Endpoints**:  
A API foi projetada para gerir os seguintes recursos principais: Users, Clubs, Courts e Rentals, nos quais cada recurso possui endpoints dedicados para operações de criação, leitura, atualização e exclusão, permitindo interações eficientes com o sistema.

**Segurança**:  
A API utiliza o esquema de autenticação Bearer Token para proteger os endpoints. Apenas utilizadores autenticados podem aceder a informações sensíveis, como os detalhes de um clube específico ou as reservas de rentals.

**Paginação e Filtros**:  
Embora a especificação atual não trate diretamente da paginação em todos os endpoints, os recursos que retornam listas (como Clubs, Courts, Rentals) podem ser facilmente adaptados para suportar a paginação caso a necessidade surja no futuro.

**Respostas e Códigos de Status**:  
As respostas da API seguem convenções comuns com códigos de status HTTP. Por exemplo, 201 para criação bem-sucedida, 400 para erros de solicitação, 401 para falha de autenticação e 404 quando o recurso solicitado não é encontrado.

**Descrição das Operações**:  
Cada operação é devidamente documentada com descrições claras sobre o que faz. Além disso, os parâmetros de entrada e saída estão especificados com tipos e exemplos, o que facilita a compreensão da utilização da API.

**Componentes Reutilizáveis**:  
A especificação utiliza componentes reutilizáveis, como esquemas de dados (UserResponse, NewUser, NewClub, etc.) e respostas comuns (400BadRequest, 404NotFound), o que facilita a manutenção e a consistência da API.

**Definição de Schemas**:  
Os dados trocados pela API estão bem definidos em esquemas (por exemplo, NewUser, UserResponse, NewRental), com propriedades e tipos claros, assegurando que as entradas e saídas sejam validadas corretamente.


### Request Details
#### 1. **Como um pedido é processado no sistema?**
Cada pedido enviado ao sistema segue um fluxo estruturado, passando pelos seguintes elementos:
1. **Entrada do Pedido (API Gateway ou Controlador)**
    - O pedido chega inicialmente à API através de um _endpoint_
    - REST definido (utilizando um _framework_ como Spring Boot, por exemplo).
    - Exemplo: um pedido `POST /clubs/3/courts` é
    - enviado ao respetivo controlador responsável
    - pela entidade _court_ (_campo_).

2. **Roteamento e Controlo**
    - O roteador encaminha o pedido para a classe ou função apropriada que lida com o recurso solicitado (por exemplo, um _Service_ ou _Controller_).
    - A camada de controlo é também responsável por capturar e processar parâmetros necessários (como `clubId` e `name`) fornecidos no corpo do pedido (_body_) ou na URL.

3. **Validação de Parâmetros**
    - Antes de executar as operações sobre o dataBase, os parâmetros fornecidos são validados para garantir que os dados são consistentes e válidos,
      de acordo com os requisitos do sistema.
    - Exemplos de validação:
        - Verificar se o `clubId` (ID do clube) existe no sistema.
        - Confirmar que o campo `name` (nome do campo) não está vazio.
        - Validar que o token fornecido no cabeçalho de autenticação (_Bearer Token_) é válido.

4. **"Services"**
    - Depois das validações, o pedido é encaminhado para a **Services**, que contém a lógica  necessária para processar o pedido.
    - No caso da criação de um Court, a lógica pode incluir:
        - Confirmar que o clube (_clubId_) existe na base de dados.
        - Garantir que o nome do campo fornecido não está duplicado no mesmo clube.
        - Guardar um novo campo no sistema.

5. **Persistência na Base de Dados (Camada de Repositórios)**
    - Após a execução da lógica de Services, os dados são armazenados ou recuperados da base de dados através de repositórios (_Repositories_) ou DAOs.
    - Exemplo:
        - `save()` é utilizado para persistir os dados(por exemplo, o método getUserById em DatabaseUserRepository.kt recupera os detalhes de um utilizador pelo seu uid).
        - `findById()` é usado para validar que um clube ou campo existe antes de avançar.

6. **Resposta ao Pedido**
    - No final, o sistema fornece uma resposta ao cliente, incluindo um código de estado HTTP adequado (ex:, `201 Created` para recursos criados, `200 OK` para consultas bem-sucedidas) juntamente com o corpo da resposta no formato JSON.

#### 2. **Classes/Funções Relevantes no Processamento do Pedido**
- **Controladores (Controllers):**
  São responsáveis por interagir diretamente com os pedidos recebidos e por mapear os _endpoints_ REST.
    - Exemplo: o _endpoint_ `POST /clubs/{clubId}/courts` pode ser associado ao método `createCourt`.

- **Serviços (Services):**
  Contêm a lógica que fica entre o _Controller_ (Controlador) e a camada de dados.
    - Exemplo: o método `createCourt()` verifica parâmetros e executa a lógica para registar um novo campo.

- **Repositórios:**
  Controlam o acesso à base de dados para armazenamento e recuperação de informações:
    - Exemplo:
        - `save()` é utilizado para guardar os dados.
        - `findById()` é implementado para verificar a existência de uma entidade antes de ações adicionais.

#### 3. **Validação de Parâmetros**
A validação de parâmetros ocorre em duas grandes etapas:
1. **Validação Inicial (Camada de Controlador):**
   Nesta fase, são validados elementos básicos, como:
    - Os tipos de dados dos parâmetros fornecidos na URL ou no corpo do pedido.
      Exemplo: se `clubId` for enviado como texto em vez de um número, o sistema devolve um erro de validação.
    - Verificação de dados obrigatórios.
      Exemplo: o campo `name` (nome do campo) é obrigatório; se não for fornecido, o sistema retorna um erro `400 Bad Request`.

2. **Validação (Services):**
   Nesta fase, validações mais complexas são realizadas, tais como:
    - Confirmar que o ID do clube fornecido (`clubId`) existe antes de proceder com a criação de um novo campo.
    - Garantir que o nome do novo campo é único dentro de um clube.



### Connection Management

### Gestão de Conexões (Connection Management)
#### 1. **Como são criadas as conexões?**
No sistema, as conexões são criadas através de um
_Connection Pool_ (piscina de conexões),
uma funcionalidade comum em ORMs (Object-Relational Mappers)
ou bibliotecas de gestão de bases de dados, como Hibernate ou
JDBC. O _Connection Pool_ é responsável por gerir um conjunto de conexões
pré-estabelecidas com a base de dados, evitando a criação e terminação
constante de conexões, o que seria ineficiente em termos de desempenho.
Funcionamento geral:
- **Na inicialização da aplicação**,
  as conexões são criadas com base nas configurações
  definidas no ficheiro de configuração 'courtsServer.kt' (ex. credenciais da base de dados).


- **Durante a execução de uma operação**, as
  conexões disponíveis no pool são reutilizadas. O _Connection Pool_
  fornece ao sistema uma conexão já existente em vez de criar uma nova.

#### 2. **Como as conexões são utilizadas?**

As conexões que vêm do _pool_ são utilizadas no
âmbito transacional do sistema. Cada operação que interage
com a base de dados é tipicamente associada a uma transação.
No caso deste projeto, o fluxo genérico seria:
1. **Início de uma Requisição**:
    - Quando uma API é chamada para uma operação que interage com a
      base de dados (ex:, `POST /clubs/3/courts`), é iniciada uma
      transação, em que ela é frequentemente gerida pela camada de
      serviço.

2. **Associação de uma Conexão à Transação**:
    - Uma única conexão do _pool_ é atribuída
      à transação. Durante a execução da transação,
      todas as operações realizadas (ex:, consultas,
      atualizações, inserções) utilizam essa conexão.
      Exemplo:
      Inserir um novo court no club:
      Primeiro, o sistema valida se o `clubId`
      existe no banco através de uma consulta SQL;
      Depois, executa a operação de inserção do
      novo court (ambas as ações partilham a mesma conexão).

3. **Confirmação ou Anulação da Transação**:
    - Ao final da operação:
        - **Se todas as ações forem concluídas com sucesso**,
          a transação é **confirmada (committed)**, confirmando
          as alterações realizadas no banco de dados.
          **Se ocorrer um erro**, a transação é **anulada**,
          garantindo que nenhuma parte da operação parcial alterou o estado
          da base de dados.

#### 3. **Como as conexões são descartadas?**
Após concluída (ou anulada) uma transação:
- A conexão utilizada é **devolvida ao Connection Pool**,
  ficando disponível para ser reutilizada por outra transação ou pedido.
- Isto garante que:
    - Não haja vazamento de conexões.
    - As conexões são reutilizadas eficientemente, melhorando o desempenho do sistema.

#### 4. **Relação com Âmbito Transacional (Transaction Scope)**
A gestão de conexões está intrinsecamente ligada ao
âmbito transacional. Eis como:
**Uma transação = Uma conexão**:
Cada transação utiliza exatamente uma conexão do _pool_.
Durante a transação, esta conexão é exclusiva e só é
partilhada quando a transação termina.

- **Âmbito da transação (Transaction Scope)**:
  No projeto, cada operação de escrita/leitura realizada
  em serviços é gerida automaticamente pelo âmbito transacional
  definido .
    - Quando um método está marcado como transacional:
        - Uma conexão é atribuída ao método no início da execução.
        - O commit ou rollback é
          gerido automaticamente ao final do
          método, dependendo do sucesso ou falha da operação.

    - Operações fora deste âmbito ficam fora da transação,
      muitas vezes ligadas apenas a consultas que não alteram
      o estado do sistema.

- **Exemplo prático (no projeto)**:
    - Endpoint `POST /clubs/3/courts`:
        1. O controlador recebe o pedido e chama o método de serviço `createCourt`.
        2. O método `createCourt` está anotado como transacional .
        3. Uma conexão do _pool_ é alugada para esta transação.
        4. São executadas as etapas da transação:
            - Validação do `clubId` no banco de dados.
            - Inserção de um novo Court.

        5. No final :
            - Sucesso: As alterações são confirmadas.
            - Falha: As ações são revertidas.

        6. A conexão é devolvida ao pool e fica disponível para novas operações.

### Benefícios da Gestão de Conexões
1. **Desempenho Melhorado**:
    - O uso de um _pool_ de conexões evita criar e fechar constantemente conexões com a base, que é um processo dispendioso em termos de recursos.

2. **Segurança e Consistência**:
    - A utilização de transações garante que operações múltiplas sejam tratadas como blocos únicos; ou tudo é concluído com sucesso, ou nenhuma parte da operação é aplicada.

3. **Eficiente Reutilização de Recursos**:
    - As conexões são recicladas dentro do sistema, reduzindo o custo de manutenção de múltiplas conexões abertas.


### Data Access

#### **Classes Criadas para Ajudar no Acesso a Dados**
No projeto, foram criadas classes para encapsular a lógica de acesso à base de dados, facilitando a interação com as tabelas. Essas classes, como por exemplo `clubDB`, `userDB`, `courtDB` e `rentalDB`, são responsáveis por realizar operações específicas sobre diferentes entidades do sistema.
Cada classe tem uma função bem definida:
- **`clubDB`** lida com informações relacionadas a clubes, incluindo a busca de dados detalhados sobre um clube e seu proprietário.
- **`userDB`** é utilizada para realizar operações na tabela de utilizadores, como recuperar o ID de um utilizador com base no nome.
- **`courtDB`** é relacionada aos Courts, verificando, por exemplo, a associação entre Courts e clubes.
- **`rentalDB`** foca em operações relacionadas aos Rentals, permitindo navegar pela associação entre Courts e clubes.
- Além disso, existe um ficheiro SQL independente (`addData.sql`) usado para preencher dados na base.

Essas classes simplificam e centralizam a lógica de acesso à base, permitindo a reutilização de métodos e facilitando a manutenção.
#### 2. **Consultas SQL Não-Triviais**
Foram identificadas algumas consultas que vão além de instruções simples, o que indica que foi realizado operaçoes mais complexas  no acesso aõs dados. Entre os exemplos mais destacados estão:
- **Consultas com junções entre tabelas**, para extrair informações detalhadas que envolvem múltiplas entidades, como obter informações sobre um clube junto com detalhes do proprietário.
- **Inserção de dados derivados de outras tabelas**, como um caso em que novos registos são adicionados baseando-se em dados existentes de outras tabelas.
- **Filtros condicionais** para verificar a existência de registos com base nos seus identificadores, assegurando a integridade antes de realizar operações mais complexas.
- **Consultas relacionadas a associações**, como identificar a que clube pertence um campo específico.

Estas consultas mostram que o projeto faz um uso eficiente e direto de SQL para gerir relacionamentos e validar dados.


### Error Handling/Processing

#### 1. **Estrutura de Tratamento de Erros**
No projecto foi criado o ficheiro `error.kt`, que desempenha um papel importante na gestão de erros.
Este ficheiro contém, classes e funções relacionadas a erros internos da aplicação, sendo ele  utilizado em diversas partes do projeto, como nos
ficheiros `clubDataMem.kt` e `courtDataMem.kt`. Este componente centralizado para erros facilita na resolução no tratamento de exceções e no fornecimento de mensagens de erro .
No contexto de erros, a aplicação inclui:
- **Erros internos:** Definidos para capturar situações associadas a falhas críticas no sistema.
- **Mensagens claras e detalhadas:** Com o objétivo de fornecer contexto suficiente  para os utilizadores durante a execução da aplicação.



Tendo o  `error.kt` como base, o tratamento de erros foi
implementado de forma a identificar situações inesperadas,
lançar exceções adequadas e, em alguns casos, interromper a
execução do fluxo lógico para evitar falhas severas na aplicação, em que são especidicádas
a seguir os principais aspétos do tratamento de erros e como
esses impactos afetam o comportamento da aplicação:
#### 1. **Uso de Exceções para Sinalizar Erros Críticos**
- Foram identificadas várias partes do código onde são lançadas exceções ao ocorrerem problemas irreversíveis, como falhas em operações na base de dados ou validações de dados inválidos. Exemplos incluem:
    - **Erros no Processo de Criação:** Se uma operação de inserção na base de dados falhar (por exemplo, nenhuma linha foi afetada por uma instrução `INSERT`), é lançado um erro para sinalizar o problema.
    - **Validações de Dados:** Caso sejam fornecidos dados inválidos (como um e-mail incorreto ou inexistência de um utilizador), exceções são lançadas para interromper o fluxo de execução.

#### 2. **Controle de Condições de Autorização**
- Existem verificações para assegurar que a autorização ou permissões de um utilizador sejam válidas. Se a autorização falhar, é lançada uma exceção, mantendo a segurança e o controle do acesso à aplicação.

#### 3. **Casos Comuns de Erros Tratados no Projeto**
- **Falhas em Operações de Banco de Dados:** Lançamento de exceções em caso de falha na criação de utilizadores, Rentals ou associações entre entidades.
- **Validação de Dados Inválidos:** Deteção de dados incorretos ou inconsistentes, como identificadores não encontrados ou dados fornecidos que não seguem os padrões esperados.
- **Erros de Autorização:** Falha no contróle de acesso devido a permissões insuficientes ou tokens inválidos.

#### 4. **Impacto no Comportamento da Aplicação**
- **Interrupção de Processos:** Quando erros críticos são identificados, o fluxo é interrompido imediatamente para evitar inconsistências ou falhas maiores.
- **Segurança e Validação:** O uso de tratamento de exceções incrementa a segurança da aplicação ao garantir que operações invalidas não sejam executadas.
- **Mensagens de Erro:** As mensagens de erro fornecem contexto útil para depuração ou comunicação com o utilizador no caso de falhas.
